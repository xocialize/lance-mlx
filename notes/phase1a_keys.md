## TL;DR — three things to know before Phase 1b

1. **All 1021 LLM key NAMES are identical between Lance_3B and Lance_3B_Video** — but **one tensor differs in SHAPE**. `latent_pos_embed.pos_embed` is `[4096, 2048]` in Lance_3B (64×64 spatial grid for images only) vs `[126976, 2048]` in Lance_3B_Video (spatial×temporal = 4096 × 31 positions for video latents). All other 1020 tensors are shape-identical. The Video variant additionally bundles the Qwen2.5-VL ViT (390 extra tensors under `vit_model.*`); Lance_3B does not (its ViT lives in the sibling `Qwen2.5-VL-ViT/vit.safetensors`). **One key-map handles both checkpoints** in `scripts/02_convert.py`; the `LatentPosEmbed` scaffold's `max_latent_size` becomes a load-time parameter rather than a hardcoded default.

2. **Three structural components are present in the safetensors that no scaffold currently models.** `vae2llm` (input projection from 48-ch VAE latents → 2048-hidden), `latent_pos_embed.pos_embed` ([4096, 2048] learned positional embedding over a 64×64 latent grid), and a second final RMSNorm `model.norm_moe_gen` (GEN-expert sibling of the standard final norm). All three must be added to `src/lance_mlx/model/` before the converter can land.

3. **Two scaffold corrections are required.** `llm2vae` has both `.weight` and `.bias` — HANDOFF.md's "bias=False" was wrong. The `TimestepEmbedder` MLP is keyed `mlp.0` / `mlp.2` (Sequential pattern) in the safetensors, not the scaffold's `proj_in` / `proj_out`. Both fixes are one-liners in `flow_head.py` and `time_embedder.py`.

## Scaffold deltas (work for Phase 1b before the converter lands)

Code changes needed in `src/lance_mlx/model/` to match the actual safetensors:

### D1. `flow_head.py` — add bias to `llm2vae`

**Current (line 47):**
```python
self.llm2vae = nn.Linear(hidden_size, latent_channels, bias=False)
```

**Required:**
```python
self.llm2vae = nn.Linear(hidden_size, latent_channels, bias=True)
```

Empirical evidence: `llm2vae.weight [48, 2048]` *and* `llm2vae.bias [48]` both present. The HANDOFF.md claim "bias=False" was based on a docstring read of upstream `lance.py` that did not match the actual saved tensors. Trust the safetensors.

### D2. `time_embedder.py` — rename to Sequential MLP pattern

**Current (lines 46–48):**
```python
self.proj_in = nn.Linear(freq_dim, hidden_size)
self.proj_out = nn.Linear(hidden_size, hidden_size)
self.act = nn.SiLU()
```

**Required:**
```python
# Match safetensors keys: time_embedder.mlp.0 (Linear), .1 (SiLU implicit), .2 (Linear)
self.mlp = nn.Sequential(
    nn.Linear(freq_dim, hidden_size),
    nn.SiLU(),
    nn.Linear(hidden_size, hidden_size),
)
```

Empirical evidence: `time_embedder.mlp.0.weight [2048, 256]`, `time_embedder.mlp.0.bias [2048]`, `time_embedder.mlp.2.weight [2048, 2048]`, `time_embedder.mlp.2.bias [2048]`. The `mlp.1` index is the SiLU (no params). Verify MLX's `nn.Sequential` produces matching key paths; if not, use an explicit `ModuleList` named `mlp` to control the indexing.

### D3. `__init__.py` — add VAE input projection module

**New file `src/lance_mlx/model/vae_bridge.py`:**

```python
"""VAE↔LLM bridge — the two thin Linears connecting Wan2.2 VAE latents
to LLM hidden states.

- `vae2llm` (Linear(48, 2048, bias=True)): VAE latent → LLM hidden (input side,
  applied to clean/noisy VAE-token embeddings before the layer stack).
- `llm2vae` (Linear(2048, 48, bias=True)): LLM hidden → velocity prediction
  at GEN positions (output side; the "flow head" of flow_head.py).

Both are independent — they do NOT share weights — but they're symmetric in
shape (just transposed dims).
"""
import mlx.core as mx
import mlx.nn as nn


class VAEInputProjection(nn.Module):
    def __init__(self, latent_channels: int = 48, hidden_size: int = 2048):
        super().__init__()
        self.vae2llm = nn.Linear(latent_channels, hidden_size, bias=True)

    def __call__(self, x: mx.array) -> mx.array:
        """x: (B, T_vae, 48) → (B, T_vae, 2048)"""
        return self.vae2llm(x)
```

Tensor keys: `vae2llm.weight [2048, 48]` + `vae2llm.bias [2048]`.

### D4. New module: latent positional embedding

**New file `src/lance_mlx/model/latent_pos_embed.py`:**

```python
"""Learned positional embedding over the VAE latent grid.

Verified 2026-05-20: single tensor `latent_pos_embed.pos_embed` of shape
[4096, 2048] — equals `max_latent_size² × hidden_size` with the shipped
`--max_latent_size 64`. Encodes spatial position in the 64×64 VAE-latent
grid. Added into the LLM input stream alongside word embeddings AND the
timestep embedding at noisy-VAE-token positions.
"""
import mlx.core as mx
import mlx.nn as nn


class LatentPosEmbed(nn.Module):
    def __init__(self, max_latent_size: int = 64, hidden_size: int = 2048):
        super().__init__()
        self.pos_embed = mx.zeros((max_latent_size * max_latent_size, hidden_size))

    def __call__(self, positions: mx.array) -> mx.array:
        """positions: (B, T_vae) int — flat indices into the latent grid."""
        return self.pos_embed[positions]
```

Tensor key: `latent_pos_embed.pos_embed [4096, 2048]`.

### D5. `lance_llm.py` — second final RMSNorm (per-expert)

The `LanceModel` docstring mentions per-layer QK norms × 36 = 144, but the **final** RMSNorm is also duplicated. Both `language_model.model.norm.weight` (UND) AND `language_model.model.norm_moe_gen.weight` (GEN) are present, each [2048].

In `LanceModel.__init__`:

```python
self.norm     = nn.RMSNorm(config["hidden_size"], eps=config["rms_norm_eps"])  # UND
self.norm_moe_gen = nn.RMSNorm(config["hidden_size"], eps=config["rms_norm_eps"])  # GEN
```

Applied at the end of the layer stack, routed by `position_group` per-token (UND tokens → `self.norm`, GEN tokens → `self.norm_moe_gen`).

### D6. Key-mapping convention — `language_model.` prefix

Every LLM tensor in the safetensors is prefixed with `language_model.` (this is mlx-vlm and the upstream HF convention for multimodal models). When `scripts/02_convert.py` translates HF safetensors → MLX module tree, the prefix needs to be stripped (because in our MLX tree, `LanceModel` is the root for the LLM, not nested under a `LanguageModel` wrapper). Alternatively, our MLX tree could mirror the HF nesting; choice is a Phase 1b design call.

## Architectural notes (new findings beyond HANDOFF.md)

These are facts we now know that weren't in the HANDOFF verified-findings section:

1. **VAE↔LLM bridge is bidirectional.** The original handoff knew about `llm2vae` (output flow head). The safetensors reveals a symmetric `vae2llm` input projection. Total bridge params: ~200K — negligible, but architecturally significant: VAE latents do NOT enter the LLM via the same `embed_tokens` path as text — they go through their own `vae2llm` Linear, then are inserted into the token stream at the GEN positions.

2. **Latent positional embedding is LEARNED, not RoPE-derived.** A single `latent_pos_embed.pos_embed [4096, 2048]` tensor — 8.4M params — encodes 2D spatial position over the 64×64 VAE-latent grid. This is *additional* to MaPE re-anchoring and Qwen2.5-VL's 3D RoPE; the three position signals coexist for VAE tokens (RoPE for sequence position, MaPE for modality offset, latent_pos_embed for spatial-within-modality).

3. **The "final norm" is per-expert.** UND tokens flow through `model.norm` before `lm_head`; GEN tokens flow through `model.norm_moe_gen` before `llm2vae`. The 144-RMSNorm count in HANDOFF was only the 36-layer × 4 q/k norms; the final-norm duplication adds 2 more (total 146 RMSNorms).

4. **Lance_3B_Video bundles the ViT inside its safetensors.** This was unexpected — HANDOFF.md and the HF repo listing both treat `Qwen2.5-VL-ViT/vit.safetensors` as a separate file (and it does exist; 1.3 GB). But `Lance_3B_Video/model.safetensors` *also* contains a full ViT under `vit_model.*` (390 tensors, ~900 M params). Worth checking whether (a) the bundled ViT is byte-identical to the separate file (then ignore the bundle), or (b) it's a fine-tuned variant that should override the separate file for video tasks. Quick win for Phase 1b: pull the separate `Qwen2.5-VL-ViT/vit.safetensors` header and diff shape-by-shape.

5. **The `_moe_gen` suffix is consistent and total.** Every duplicated component uses exactly `_moe_gen` (not `_gen` or `_moe` or `.gen.` etc.). The converter can implement the UND/GEN split as a literal string replace: `key.replace("_moe_gen", "") → UND key`. Symmetric for the reverse.
